CSUMB Capstone Project

Iperf Parser for CPUC Network Testing Data

 

Abstract:

This project facilitates the parsing and analysis of network performance testing data, provided by the California Public Utilities Commission in text files.

The text files, containing the network testing information, are passed to a Python script, which utilizes the Python objects included created by the source files provided in this project.

In a Python script, the Python object is imported. If a text file is from a Field Test, then FieldTest_File.py must be used. Given a proper text file, initialization will return an object that can be used for a number of analytical functions. Improper files return an error. The basic text file format and Python object structure is expanded upon below.

 

 

General/Expected Usage:

All of the Python files provided include a Python class. It is expected that user will call

  from FieldTest_File import FieldTest_File

This will allow the user to create a FieldTest_File object by passing the constructor the location of a text file, i.e.

  file = FieldTest_File("/home/user/files/file0001.txt")

The Python variable file will now contain an object, which has a variety of functions and local variables that can be searched for the required information.

 

In general, files provided are Field Test files (which are generated by testers that have been contracted by the CPUC to run the necessary tests). However, there are some Crowd Source generated files. These files must be passed to a separate class, which is imported as such...

  from CrowdSource_File.py import CrowdSource_File

 

Text File Format:

All files passed to the constructor are expected to be text files, containing utf-8 characters. They are also expected to be in this generic format.

 

HEADER INFORMATION

-     Date, Time, Location ID, GPS Location

-     Network Carrier, Device ID, OS and Java version info

INITIAL PING TEST

-      Checking that the West Amazon Server can be reached

-      If this fails multiple times, then no further tests are performed, and the file ends

TEST # 1 – TCP Test to West Amazon Server

This test uploads and downloads a file from the West Amazon server (***REMOVED***) over a TCP connection, measuring the throughput of the connection every second (instantaneous throughput), and then averages the speeds measured (average throughput).

-     In each test, four connections are made to the server. The host (the device being used) opens the connections on four separate ports, and all connections connect to the same receiver port of the Amazon server.

o   [  3] local 10.20.30.40 port 50076 connected with
      ***REMOVED*** port 5005

-     The four 'threads' first upload a file to the server, close the connection, create a new connection, and then download a file from the server.

o   Open, UPLOAD, Close, Open, DOWNLOAD, Close

-     At each second interval, the thread's instantaneous throughput is recorded as such...

o  [  3]  2.0-3.0 sec   272 KBytes  2228 Kbits/sec

o  The first field (within the square brackets) is the thread number. Generally, the thread numbers range from 3 to 6.

o  The second field is the time interval. In this example, the time was between the 2nd and 3rd second of the total download time.

o  The third field is the amount of data sent in this time interval (in KiloBytes).

o  The fourth field is the throughput of the connection (in Kilobits per second).

-     A final measurement contains the total time to download/upload the file, the total amount of data received/sent, and the average throughput.

o   [  3]  0.0-10.3 sec  5120 KBytes  4068 Kbits/sec

-     There are a number of possible errors within this test. If any of these errors occur, the test is ignored during analysis.

o  The test could fail halfway through, resulting in an unfinished upload or download.

o  An improper number of download threads could be started.

o  The instantaneous throughputs could all show 0 Kbits/sec, while the summary measurements could should speeds upward of 10,000,000 Kbits/sec.

o  The file could abruptly end.

o  The test could be quit by the user.

TEST # 2 – TCP Test to East Amazon Server

-     This test follows the same pattern as Test # 1, only it has been connected to the East Amazon Server (***REMOVED***).

-     In general, the throughput measured in tests to the East server will be slower than the throughput measured in the tests to the West server.

TEST # 3 – PING Test to East Amazon Server

This test sends a series of pings to the East Amazon Server (***REMOVED***), which measures the Round Trip Time (in milliseconds) from the device to the server. The test is formatted in one of two ways, but both have the same general structure.

-     The test sends a sequence of ten pings to the East Amazon Server. The packets sent return, giving the RTT of the connection

o  64 bytes from ***REMOVED***: icmp_seq=1 ttl=40 time=118 ms

-     The test concluded with a statistics summary, which includes the number of packets sent, received, and lost. It also contains the minimum, average, and maximum RTT of the packets received.

o  --- ***REMOVED*** ping statistics ---
10 packets transmitted, 10 received, 0% packet loss, time 9017ms
rtt min/avg/max/mdev = 108.704/160.958/232.360/35.203 ms

 

-     There are a number of possible errors within this test. If any of these errors occur, the test is ignored during analysis.

o  The test could fail halfway through.

o  The network could be unreachable, resulting in no packets received

o  The file could abruptly end.

o  The test could be quit by the user.

TEST # 7 – Three UDP Tests to West Amazon Server

These tests upload a file to the West Amazon server (***REMOVED***) over a UDP connection, measuring the throughput of the connection every second (instantaneous throughput), and then averages the speeds measured (average throughput).

This test is separate from Test # 9 because all three tests in this section upload data for one second. The test in # 9 is a five second test.

-     The format is very similar to the TCP tests (See Test # 1), but include a UDP specific piece of information.

-     Every UDP tests ends with a Server Report, which includes UDP specific information such as Jitter, and Datagram Loss.

-     [  3]  0.0-1.0 sec  11.0 KBytes  92.5 Kbits/sec  11.497 ms    0/   51 (0%)

o   The first field is the thread number

o   The second field is the total time to upload

o   The third field is the amount of data sent (in KiloBytes)

o   The fourth field is the average throughput (in Kilobits per second)

o   The fifth field is the UDP Jitter

o   The sixth field is the number of datagrams lost over the total number sent, followed by the percentage lost.

-     This test lasts for one second, and is repeated three times before moving on to the next test.

 

-     There are a number of possible errors within this test. If any of these errors occur, the test is ignored during analysis.

o  The test could fail halfway through, resulting in an unfinished upload.

o  The file could abruptly end.

o   The test could be quit by the user.
